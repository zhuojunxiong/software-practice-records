## 一、本次对话背景

在 v0.4.1 版本中，系统已经完成了界面和核心交互链路的整理，但 AI 功能仍然停留在占位状态。写作页和后台虽然有 AI 入口，但点击后只会提示“AI 功能将在后续版本开放”，不能真正生成摘要、推荐标签、润色正文或进行文章问答。

本次对话的目标是开发 v0.5，重点是：

1. 打通真实 AI 接口；
2. 将 AI 能力接入写作页；
3. 支持 DeepSeek API；
4. 补充 API Key 使用说明；
5. 修复 AI 聊天框过长的问题；
6. 对项目核心功能进行一次基础验收。

## 四、DeepSeek 接入过程

### 4.1 用户询问是否需要自己申请 API Key

用户问：

> 我要自己去申请 apikey 吗，怎么做？

AI 助手说明：

1. API Key 需要用户自己申请；
2. API Key 绑定账号、额度和计费信息；
3. 不能写死在项目里；
4. 应通过环境变量传入。

### 4.2 用户询问能否使用 DeepSeek

用户问：

> 能不能用 deepseek？

AI 助手确认 DeepSeek 提供 OpenAI 兼容接口，因此当前服务层可以适配 DeepSeek。

最初建议的配置方式为：

```bash
export AI_API_KEY="你的 DeepSeek API Key"
export AI_BASE_URL="https://api.deepseek.com"
export AI_MODEL="deepseek-v4-flash"
```

后续项目默认配置也调整为：

```python
AI_BASE_URL = "https://api.deepseek.com"
AI_MODEL = "deepseek-v4-flash"
AI_MAX_TOKENS = 800
```

这样以后只需要配置 `AI_API_KEY` 即可启动 AI 功能。

## 五、API Key 安全问题

在对话中，用户不小心把真实 API Key 发给了 AI 助手。

AI 助手立即提醒：

1. 真实 Key 不应该发到聊天中；
2. 已泄露的 Key 建议立即删除或重置；
3. 重新生成新的 Key 后，只在本地终端配置；
4. 不要写入 `config.py`；
5. 不要写入 README 或任何文档；
6. 不要提交到 Git。

本次项目中没有把用户提供的 Key 写入任何代码或文档。

这一点也被补充进了 `docs/deepseek-ai-setup-guide.md`。

## 七、AI 聊天框过长问题修复

### 7.1 问题现象

用户在真实使用后发现：

> AI 写作会把聊天框弄得很长。

截图显示右侧 AI 面板被长回复不断撑高，导致页面滚动区域过长，影响写作体验。

### 7.2 原因分析

AI 助手分析后认为问题主要在前端布局：

1. AI 消息区使用 grid 布局；
2. 消息列表没有固定高度；
3. grid 子项缺少 `min-height: 0`；
4. 长回答会撑开整个右侧面板；
5. 右侧面板再继续撑开页面高度。

### 7.3 修复方案

修改 `app/static/css/v041.css`：

1. 右侧 AI 面板设置固定视口高度；
2. 消息列表内部滚动；
3. 给消息区增加 `min-height: 0`；
4. 给长文本增加自动换行；
5. 移动端取消固定高度，保持自然布局。

同时修改 AI 提示词和配置：

1. 问答默认控制在 300 字以内；
2. 新增 `AI_MAX_TOKENS=800`；
3. 避免 AI 一次输出过长。

## 十二、AI 协作反思

本次 AI 协作比较典型地体现了“先实现，再验证，再修正”的过程。

### 1. AI 辅助的有效部分

AI 助手有效完成了：

1. 快速定位原有 AI 占位模块；
2. 保持现有 Flask + Jinja2 架构，不做无关迁移；
3. 复用已有 `AiLog` 表；
4. 将真实 AI 调用封装到服务层；
5. 为 DeepSeek 增加默认配置；
6. 写出从 0 使用指南；
7. 根据用户截图定位并修复前端布局问题；
8. 进行基础路由和接口测试。

### 2. 需要人工确认的部分

以下内容仍然需要用户或开发者人工确认：

1. DeepSeek API Key 是否有效；
2. 账号是否有余额或额度；
3. 实际 AI 回复质量是否符合课程演示需求；
4. 写作页视觉效果是否符合用户体验；
5. 是否需要进一步增加 AI 功能按钮，例如大纲生成、标题建议等。

### 3. 本次对话中的安全教训

用户曾将真实 API Key 发到聊天中。这个过程说明：

1. 开发 AI 功能时，Key 管理是必须强调的安全点；
2. 文档中必须写清楚“不要把 Key 发给别人”；
3. AI 助手不能把 Key 写入代码；
4. 泄露后应立即删除旧 Key 并重新生成。

这部分内容可以作为实践报告中“AI 协作与安全意识”的案例。

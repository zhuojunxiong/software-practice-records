## 五、第五阶段：写作页任务

4. 版本管理的作用会被削弱。

## 六、第五阶段完成与 SQLite readonly 排查

### 用户

第五阶段写作页已完成，没有改数据库结构，没有接入真实 AI，也没有引入新依赖。它实现了 `/write`、`/write/<id>`、旧编辑路由兼容、保存草稿、发布文章、AI 占位和未保存离开提醒。但报告中提到正式 SQLite 写入会出现 readonly。接下来是不是第六阶段？

### AI

原则上下一步是第六阶段，但在进入第六阶段之前，应该先处理 SQLite readonly 风险。

原因是写作页的核心链路是保存草稿和发布文章，如果正式数据库不能写入，写作页就不能算完全闭环。

建议先让 AI 编程助手做最小排查：

1. 检查 SQLite 数据库路径是否正确；
2. 检查 Flask 配置是否正确；
3. 检查数据库文件是否只读；
4. 检查数据库目录是否可写；
5. 判断是否是测试沙箱环境导致；
6. 确认本机真实运行环境能否保存草稿和发布文章。

排查结果显示：`readonly` 是 AI 编程助手沙箱测试环境限制，不是项目配置或数据库本身的问题。本机真实权限下 `/write` 保存草稿、编辑保存、发布文章均可成功写入。

因此，该问题不需要修改数据库结构，也不影响 v0.4.1 继续推进。

## 七、关于直接打开 Jinja 模板文件的问题

### 用户

我打开写作页后看到一堆模板代码，页面很奇怪。怎么第五阶段之后变成这样了？

### AI

这不是第五阶段把前端做坏了，而是打开方式不对。

用户当时是直接用浏览器打开了模板文件，例如：

```text
项目目录/app/templates/user/write.html
```

但这个文件是 Jinja2 模板，不能直接双击打开。模板中的：

```jinja2
{% extends "base.html" %}
{% block content %}
{{ ... }}
```

必须经过 Flask 服务器渲染后，才会变成真正的 HTML 页面。

正确打开方式是先启动 Flask 项目，再访问：

```text
http://127.0.0.1:5000/write
```

如果直接打开模板文件，浏览器只会把 Jinja 模板源码当成普通文本显示，样式和逻辑都不会正常加载。

## 十、实机测试后发现的发布问题

### 用户

我注册了一个新账号，写了一篇文章，但是点击发布只能保存为草稿，不能真正发布。是不是数据库有问题？

### AI

这不是数据库问题，而是写作页前端 JavaScript 的表单提交细节问题。

原本“发布文章”按钮依赖：

```html
name="status"
value="published"
```

提交给后端。

但前端 JS 在提交表单时，为了显示“发布中”状态，把按钮设置为了 `disabled`。浏览器规则中，`disabled` 的按钮不会随表单提交。因此后端收不到：

```text
status=published
```

就会按默认逻辑保存为草稿。

修复方式是：

1. 在按钮禁用前，读取当前按钮的 `name` 和 `value`；
2. 将它们复制到隐藏 input；
3. 再禁用按钮；
4. 确保后端能收到 `status=published`。

涉及文件：

```text
app/static/js/v041.js
```

修复后，文章可以正常发布，状态为 `published`，并跳转到文章阅读页。

## 十一、实机测试后发现的清谈问题

### 用户

为什么清谈功能不能正常使用？我无法在文章下面发表评论。

### AI

这也不是数据库问题，而是 v0.4.1 改版时把“评论输入入口”拿掉了。

之前阅读页按“安静阅读”设计，弱化或隐藏了评论区；而 `/talk` 只是展示交流信息，没有提供从某篇文章发起清谈的入口。

后端评论路由仍然存在，问题是前端链路没有打通。

最小修复方式：

1. 在文章阅读页右侧增加“发起清谈”入口；
2. 点击后进入：

```text
/talk?article=文章slug
```

3. `/talk` 顶部出现“发起清谈”表单；
4. 提交后复用现有评论系统；
5. 评论状态为 `pending`；
6. 管理员审核后才公开显示。

涉及文件：

```text
app/user/routes.py
app/templates/v041/article_reading.html
app/templates/user/talk.html
```

修复后，文章页可以发起清谈，评论进入待审核状态，符合轻量交流中心定位。

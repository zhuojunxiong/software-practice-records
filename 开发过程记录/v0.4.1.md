# v0.4.1 版本迭代总结文档

> 文档类型：版本迭代总结
> 项目类型：多用户知识专栏博客系统
> 当前版本：v0.4.1
> 技术栈：Flask + Jinja2 + Flask-Login + SQLAlchemy + SQLite
> 版本主题：界面与核心交互链路重构
> AI 状态：仅保留占位，不接入真实 AI
> 前端框架状态：未迁移 Vue / React，继续使用服务端渲染

---

## 一、版本背景

在前几个版本中，项目逐渐从一个基础博客系统，调整为一个面向课程实践的 **多用户知识专栏博客系统**。

早期版本主要解决了项目能不能运行、功能边界是否清楚、系统是不是单人博客等问题。随着需求逐渐明确，项目定位从“单人个人博客”修正为“每个用户都可以创建专栏、写博客、阅读他人内容并进行交流”的多人知识创作平台。

v0.4 阶段主要解决前端界面过于简陋的问题，将项目从“能运行的课程 Demo”推进到“视觉上更完整、更现代、更适合答辩展示”的产品化界面阶段。

v0.4.1 则是在 v0.4 的基础上继续收口，不再只是局部美化页面，而是把认证、个人空间、写作页、作者主页、专栏页、状态页和版本文档全部补齐，形成一个相对完整、稳定、可演示的版本。

---

## 二、版本定位

v0.4.1 的定位是：

> 在现有 Flask + Jinja2 架构下，完成界面、认证、个人空间、写作链路、补充页面和状态页的统一重构，使系统具备稳定演示能力。

本版本不是技术栈迁移版本，也不是 AI 功能接入版本。

本版本重点解决：

1. 登录注册页面不统一；
2. 登录后缺少完整个人空间；
3. 写作页仍是旧页面或旧逻辑；
4. 作者主页、专栏详情页风格不统一；
5. 403 / 404 / 500 状态页较粗糙；
6. 清谈功能链路不完整；
7. 发布文章流程存在真实使用问题；
8. 缺少 v0.4.1 版本说明文档；
9. 项目需要形成可以提交、演示和继续迭代的稳定节点。

## 四、v0.4.1 与 v0.4 的区别

v0.4 更偏向前端 UI/UX 产品化改造，主要目标是让页面不再像默认 HTML 页面，形成统一视觉系统。

v0.4.1 则更进一步，把“页面好看”扩展为“页面链路完整”。

两者区别可以概括为：

| 版本     | 重点          | 说明                         |
| ------ | ----------- | -------------------------- |
| v0.4   | 前端 UI/UX 改造 | 解决页面丑、风格不统一、像默认网页的问题       |
| v0.4.1 | 核心链路收口      | 补齐认证、个人空间、写作页、状态页和最终验收     |
| v0.5   | 后续建议        | 整理 JSON API，为 AI 和前后端分离做准备 |
| v1.0   | 后续建议        | 接入轻量 AI，验证 AI 产品逻辑         |
| v2.0   | 后续建议        | 再考虑 Vue / React 等现代前端架构    |
| v3.0   | 后续建议        | 做高级 AI、语义搜索、知识库等能力         |

## 五、总体开发策略

v0.4.1 没有一次性大重构，而是采用分阶段推进方式。

这种方式符合软件工程中的迭代交付思想：

1. 每一阶段有明确边界；
2. 每一阶段只做一类链路；
3. 每一阶段完成后自测；
4. 后一阶段不破坏前一阶段；
5. 最后进行全量验收；
6. 发现问题后做最小修复。

v0.4.1 共分为六个阶段：

1. 第一阶段：视觉系统与 shared 模板基础；
2. 第二阶段：公共阅读链路；
3. 第三阶段：认证链路；
4. 第四阶段：个人空间链路；
5. 第五阶段：写作页；
6. 第六阶段：补充页、状态页和版本文档。

## 十三、最终实机修复一：发布文章只能保存为草稿

第六阶段完成后，使用真实账号和真实文章进行本机测试时，发现点击“发布文章”后文章没有真正发布，而是仍然保存为草稿。

### 问题现象

1. 用户进入 `/write`；
2. 输入文章标题和内容；
3. 点击“发布文章”；
4. 页面提交成功；
5. 文章状态仍然是 `draft`；
6. 没有变成 `published`。

### 问题原因

该问题不是数据库问题，也不是后端发布逻辑问题，而是前端表单提交细节问题。

原本发布按钮依赖：

```html
name="status"
value="published"
```

向后端提交发布状态。

但是写作页 JavaScript 在提交表单时，为了显示“发布中...”状态，把提交按钮设为了 `disabled`。

浏览器表单规则中，`disabled` 的表单控件不会随表单一起提交，因此后端收不到：

```text
status=published
```

后端只能按默认逻辑将文章保存为草稿。

### 修复方式

在 `app/static/js/v041.js` 中做最小修复：

1. 在禁用按钮前，读取当前点击按钮的 `name` 和 `value`；
2. 创建或更新隐藏的 `hidden input`；
3. 将 `status=published` 写入隐藏字段；
4. 再禁用按钮并提交表单；
5. 确保后端能收到发布状态。

### 修复结果

修复后已验证：

1. 后端可以收到 `status=published`；
2. 文章状态可以变成 `published`；
3. `published_at` 可以正常写入；
4. 发布成功后跳转 `/articles/<slug>`；
5. 不需要改数据库；
6. 不需要改模型；
7. 不影响保存草稿逻辑。

### 涉及文件

```text
app/static/js/v041.js
```

## 十四、最终实机修复二：文章页无法发起清谈

实机测试中还发现：用户在文章详情页下无法发表评论或发起清谈。

### 问题现象

1. 用户发布文章后进入文章阅读页；
2. 页面没有明显评论入口；
3. 用户无法直接围绕该文章发起交流；
4. `/talk` 页面只展示交流内容，没有从具体文章进入的提交入口。

### 问题原因

v0.4.1 前面为了保持“安静阅读”体验，阅读页弱化或移除了评论输入区域。

但 `/talk` 只是轻量交流中心，本身没有完整承担“从某篇文章发起评论”的入口。

因此问题不在数据库，也不在评论后端，而在前端链路缺失：

```text
文章详情页 → 发起清谈 → 提交评论 → 待审核
```

这个链路没有打通。

### 修复方式

采用最小修复：

1. 在文章阅读页右侧增加“发起清谈”入口；
2. 点击后跳转：

```text
/talk?article=文章slug
```

3. `/talk` 页面根据 `article` 参数展示对应文章；
4. `/talk` 顶部显示“发起清谈”表单；
5. 提交后复用现有评论系统；
6. 评论状态为 `pending`；
7. 管理员审核后评论才公开显示。

### 修复结果

修复后已验证：

1. 文章页显示“发起清谈”入口；
2. `/talk?article=...` 能显示对应文章；
3. 清谈表单可以提交；
4. 提交后成功创建评论；
5. 评论状态为 `pending`；
6. 没有新增消息表；
7. 没有实现复杂 IM；
8. 没有破坏 `/talk` 轻量交流中心定位。

### 涉及文件

```text
app/user/routes.py
app/templates/v041/article_reading.html
app/templates/user/talk.html
```

## 十八、当前仍存在的限制

### 1. 阅读历史仍是轻量兜底

当前没有新增真实 `ReadingHistory` 表，阅读轨迹主要用收藏数据兜底。

后续可以在 v0.5 或 v1.0 之后设计真实阅读历史。

### 2. 清谈仍是轻量版

当前清谈基于评论系统实现，不是即时聊天。

它能满足文章交流的基本需求，但没有实现：

1. 私信；
2. 实时聊天；
3. WebSocket；
4. 消息通知；
5. 复杂会话系统。

### 3. AI 仍是占位

写作页右侧 AI 问答区只是占位。

后续如要接入真实 AI，需要单独设计：

1. `/api/ai/*`；
2. API Key 管理；
3. 请求超时；
4. 错误兜底；
5. 使用日志；
6. 内容安全边界。

### 4. 架构仍是 Flask + Jinja2

当前架构适合课程项目、本机演示和快速迭代，但还不是长期大型产品架构。

后续如果要继续做大，可以在 API 稳定后再迁移到 Vue / React。

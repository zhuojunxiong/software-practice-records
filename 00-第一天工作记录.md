# 第一天工作记录

## 一、初始状态

一开始拿到周报模板和参考资料后，直接尝试编写周报和项目立项计划。但在整理过程中发现，参考资料、周报内容、功能结构和开发计划之间存在明显混乱：

- 功能目标不清晰；
- 结构和开发计划交叉；
- 部分内容不满足提交要求；
- 缺少后续模块化更新的空间。

因此，不能把“写周报”作为项目推进的主线。周报应当是项目过程的记录，而不是项目开发的驱动力。

## 二、重新梳理后的推进逻辑

后续应先完成项目逻辑，再把过程整理进周报和报告中。整体步骤如下：

1. 理解实训总体任务和课程安排。
2. 全面了解已有代码框架和个人博客系统示例。
3. 查询资料，分析痛点和可引入的 AI 辅助点。
4. 总结项目的真实框架和项目计划，按模块化方式安排开发步骤。
5. 编写项目开发文档和第一周周报。
6. 完成需求分析后，进入软件分析与设计阶段。
7. 将需求具体化，查询可满足需求的技术栈，并整合成可执行的软件设计。
8. 进行功能模块化编码。
9. 整体跑通系统并进行软件测试。
10. 优化可改进功能，并检验模块化设计在产品中的实际效果。

## 三、关键反思

需求分析阶段和具体实现阶段不能混在一起，否则会导致项目定义不清晰。实训本身只是课程展示形式，真正重要的是完成一个有需求、有逻辑、能运行、能说明的软件作品。

如果项目功能和市面已有产品高度重合，就需要思考它在当前实训中的具体价值。更可行的方向是：先选择一个小切入点，完成一个可运行的 Demo，后续再继续扩展。

## 四、当前结论

本项目应以“个人博客系统”为基础，但不能只停留在传统博客功能堆叠上。更合理的方向是做一个具备内容记录、整理、展示、检索和 AI 辅助能力的个人内容系统原型。

## 五、第一天代码实现口径

在完成前期方向梳理后，第一天进一步明确了代码实现的第一版边界。项目不能被理解成“随便做一个博客网站”，而应作为一个可运行、可演示、可扩展、符合软件工程思想的个人博客系统 V0.1 来推进。

第一版的核心目标不是追求功能数量，而是先做出一个结构完整、主流程稳定、便于后续扩展的最小可用版本。系统定位为个人内容记录、整理、展示和检索系统，前台用于游客浏览文章，后台用于管理员管理文章、分类、标签和评论，并预留 AI 辅助能力，用于后续实现文章摘要生成、标签推荐和 AI 协作记录。

## 六、V0.1 技术路线

第一版采用更适合快速落地和本地演示的技术路线：

```text
Python
Flask
Flask-Login
SQLAlchemy
SQLite
Jinja2
Bootstrap
Markdown 可选
```

这样选择的原因是：

1. SQLite 方便本地运行和课程答辩，不依赖复杂数据库环境。
2. SQLAlchemy ORM 能体现规范的数据模型设计，避免大量原生 SQL 堆叠。
3. Jinja2 + Bootstrap 能快速完成前后台页面，不做前后端分离，降低联调成本。
4. Flask Blueprint 适合按模块拆分代码，方便后续扩展。
5. AI 功能第一版只做 Mock，不调用真实 AI 接口，避免接口不稳定影响演示。

## 七、V0.1 功能边界

### 7.1 第一版必须完成的主流程

```text
启动项目
-> 初始化数据库
-> 首页展示已发布文章
-> 查看文章详情
-> 按分类浏览文章
-> 按标签浏览文章
-> 关键词搜索文章
-> 管理员登录后台
-> 管理员新增文章
-> 管理员编辑文章
-> 管理员删除文章
-> 管理员管理分类
-> 管理员管理标签
-> 游客提交评论
-> 评论默认待审核
-> 管理员审核评论
-> 审核通过的评论显示在前台
-> 后台预留 AI 摘要和 AI 标签推荐接口
-> AI 操作记录保存到日志表
```

### 7.2 前台游客功能

游客不需要注册登录。第一版前台必须实现：

1. 首页文章列表：只展示已发布文章，显示标题、摘要、发布时间、分类、标签和浏览量。
2. 文章详情页：显示标题、摘要、正文、分类、标签、发布时间和浏览量，每次访问详情页浏览量自动加 1。
3. 分类浏览：点击分类后，显示该分类下的已发布文章。
4. 标签浏览：点击标签后，显示该标签下的已发布文章。
5. 关键词搜索：支持按标题、摘要或正文关键词搜索文章。
6. 评论提交：游客可以提交昵称、邮箱和评论内容，评论默认进入待审核状态，不能直接显示在前台。

### 7.3 管理员后台功能

管理员登录后才能访问后台。第一版后台必须实现：

1. 管理员登录与退出登录。
2. 默认管理员账号为 `admin`，默认密码为 `admin123`。
3. 密码必须哈希保存，不能明文保存。
4. 后台首页显示文章数量、分类数量、标签数量、评论数量等基础统计。
5. 文章管理支持列表、新增、编辑、删除、草稿、发布、下架、选择分类和绑定多个标签。
6. 分类管理支持新增、编辑、删除和列表查看。
7. 标签管理支持新增、编辑、删除和列表查看。
8. 评论管理支持查看评论、查看待审核评论、审核通过评论和删除评论。

### 7.4 AI Mock 功能

第一版 AI 功能只做 Mock，用于先把模块结构和数据记录设计好：

1. 设计 `AIService` 抽象接口。
2. 实现 `MockAIService` 模拟服务。
3. 根据文章正文生成简单摘要。
4. 根据文章正文推荐 3 到 5 个标签。
5. 保存 AI 操作日志。
6. 后台可以查看 AI 日志。
7. AI 输出不能默认直接采用，必须保留人工确认或修改空间。

### 7.5 明确不做的功能

第一版禁止加入以下功能，避免项目失控：

1. 普通用户注册登录。
2. 点赞、收藏、关注、私信。
3. 复杂社区功能。
4. Vue 或前后端分离架构。
5. 真实 AI 接口调用。
6. 把标签直接存成字符串。
7. 评论提交后直接显示。
8. 明文保存密码。
9. 把所有功能写在一个文件中。
10. 引入过多不必要依赖。

## 八、数据库模型设计

第一版至少包含以下 7 类数据模型：

### 8.1 User 管理员表

字段：

```text
id
username
password_hash
nickname
role
status
created_at
updated_at
```

要求：

1. `password_hash` 不能保存明文密码。
2. 第一版只需要管理员，不做普通用户注册。

### 8.2 Article 文章表

字段：

```text
id
title
summary
content
category_id
status
view_count
created_at
updated_at
published_at
```

要求：

1. `status` 包括 `draft`、`published`、`archived`。
2. 只有 `published` 状态的文章显示在前台。

### 8.3 Category 分类表

字段：

```text
id
name
description
sort_order
created_at
```

要求：

1. 一个分类可以对应多篇文章。
2. 一篇文章只属于一个分类。

### 8.4 Tag 标签表

字段：

```text
id
name
created_at
```

### 8.5 ArticleTag 文章标签关联表

要求：

1. 文章和标签是多对多关系。
2. 不允许把标签简单存成文章表里的字符串。

### 8.6 Comment 评论表

字段：

```text
id
article_id
nickname
email
content
status
created_at
```

要求：

1. `status` 包括 `pending`、`approved`、`deleted`。
2. 评论提交后默认 `pending`。
3. 前台只展示 `approved` 评论。

### 8.7 AiLog AI 日志表

字段：

```text
id
article_id
scene
input_text
ai_output
adopted_result
is_adopted
problem_found
created_at
```

要求：

1. `scene` 可以是 `summary`、`tag`、`outline` 等。
2. 必须记录 AI 输入、AI 输出、是否采用、人工修改说明。
3. 方便后续写入 AI 协作记录和答辩材料。

## 九、项目目录结构

第一版代码不应全部写在一个 `app.py` 中，而应使用模块化结构：

```text

│   ├── admin/

│   │   └── decorators.py
│   ├── article/

│   │   ├── forms.py

│   │   └── prompts.py
│   ├── dashboard/

│       ├── css/

│   └── demo_data.py
├── tests/
├── requirements.txt
├── config.py
├── run.py
└── README.md
```

模块职责要求：

1. 使用 Flask Blueprint 拆分模块。
2. `app/__init__.py` 只负责创建应用、初始化扩展、注册蓝图。
3. `extensions.py` 放置 `db`、`login_manager` 等扩展对象。
4. `models.py` 只放数据库模型。
5. `routes.py` 负责接收请求、调用 service、返回页面。
6. `services.py` 负责业务逻辑。
7. AI 功能单独放在 `app/ai/` 模块。
8. 评论功能单独放在 `app/comment/` 模块。
9. 后续替换 AI、搜索或评论模块时，不能影响其他模块。

## 十、第一版开发顺序

第一版不一次性生成所有功能，而是按以下顺序推进：

1. 创建项目骨架：目录结构、Flask app factory、SQLAlchemy、Flask-Login、基础页面、`requirements.txt`、`README.md` 初稿。
2. 实现数据库模型：`User`、`Article`、`Category`、`Tag`、`ArticleTag`、`Comment`、`AiLog`、初始化数据库脚本、示例数据脚本。
3. 实现前台浏览：首页文章列表、文章详情、分类筛选、标签筛选、关键词搜索、浏览量增加。
4. 实现管理员登录：登录、退出、登录保护、后台首页。
5. 实现后台内容管理：文章 CRUD、分类 CRUD、标签 CRUD、文章状态管理、文章标签绑定。
6. 实现评论审核：游客提交评论、评论待审核、管理员审核评论、管理员删除评论、前台展示已审核评论。
7. 实现 AI Mock 模块：Mock 摘要生成、Mock 标签推荐、AI 日志保存、后台查看 AI 日志。
8. 整理 README 和测试说明：补充运行说明、测试步骤和第一版功能说明。

## 十二、第一版验收标准

第一版完成后，应至少满足以下验收标准：

1. 项目可以在 Windows 本地启动。
2. 数据库可以初始化，并能插入演示数据。
3. 首页只展示已发布文章。
4. 文章详情页能打开，浏览量能自动增加。
5. 分类、标签、关键词搜索可以筛选文章。
6. 管理员可以使用默认账号登录和退出。
7. 管理员可以管理文章、分类、标签。
8. 游客评论默认待审核，不能直接在前台显示。
9. 管理员审核通过后，评论才能在前台显示。
10. AI Mock 摘要、标签推荐和 AI 日志结构可用。
11. 密码哈希保存，标签使用多对多关联表，评论具有审核状态。
12. README 中写清楚安装依赖、初始化数据库、插入演示数据、启动项目和测试步骤。

# 第三天工作记录

## 一、背景

V0.4.1 结束后，系统已经具备了稳定的演示能力：公开页面链路完整，用户可以在平台上注册、写文章、创建专栏、互动交流。但从实际使用角度看，还有三个明显问题：

1. **AI 功能全是假的**。写文章页右侧的 AI 面板只有占位提示，点击任何按钮都只显示"后续版本开放"。这在课程答辩中很难体现"AI 辅助"的定位。
2. **安全问题被忽略**。SECRET_KEY 是明文硬编码的 `dev-secret-key-change-before-production`，所有 POST 表单没有 CSRF 保护。虽然课程项目不上生产环境，但这些是基本的软件工程素养。
3. **代码存在技术债**。`datetime.utcnow()` 在 Python 3.12+ 已废弃，AI 服务用 stdlib urllib 发 HTTPS 不够稳健，文档散落且路径写死 Windows。

V0.5 的核心任务就是**打通 AI 链路**，同时**补齐安全和代码质量问题**。

## 二、V0.5 定位

> 在 V0.4.1 的稳定界面基础上，接入 DeepSeek API 实现 6 个真实 AI 写作辅助功能，并完成安全加固，使系统成为一个安全、可用、可演示的多用户知识创作平台。

三个核心原则：

1. **AI 是可配置的，不是强依赖**。不配 API Key 时系统照常运行，AI 按钮返回友好提示。
2. **安全是基础设施**。不因为是课程项目就跳过 CSRF 保护和密钥管理。
3. **代码要为未来负责**。废弃 API 要替换，不稳健的实现要重构。

## 三、AI 功能实现

### 3.1 为什么选 DeepSeek

综合考虑后选择 DeepSeek 而不是其他 AI 服务：

1. **OpenAI 兼容接口**：Chat Completions API 格式标准，后续换模型成本低。
2. **中文能力强**：摘要、润色、标签推荐都是中文任务，DeepSeek 的中文理解能力适合知识博客场景。
3. **有免费额度**：新用户注册即有额度，适合课程演示，不会因为欠费导致答辩翻车。
4. **无需特殊 SDK**：直接用 `requests` 发 HTTP POST 即可，不引入额外依赖。

默认使用 `deepseek-chat`（DeepSeek 最新旗舰模型），可通过环境变量切换到 `deepseek-reasoner` 等模型。

### 3.2 架构设计

```
写文章页 (write.html)
    │
    ├── 5 个按钮 → fetch POST → /ai/summary | tags | polish | outline | titles
    │                                   │
    └── 问答框   → fetch POST → /ai/chat  │
                                          │
                                 AIService (app/ai/services.py)
                                          │
                                 requests → DeepSeek API
                                          │
                                 AiLog → 写入 SQLite
```

每个 AI 功能独立配置 `(max_tokens, temperature)`：

| 功能 | max_tokens | temp | 说明 |
|------|-----------|------|------|
| 摘要 | 1500 | 0.3 | 低温度保证事实准确 |
| 标签 | 800 | 0.3 | 低温度保证一致性 |
| 润色 | 3000 | 0.7 | 高温度增加表达多样性 |
| 大纲 | 2000 | 0.3 | 低温度保证结构准确 |
| 标题 | 800 | 0.7 | 高温度增加创意 |
| 问答 | 2000 | 0.5 | 中等平衡准确与灵活 |

### 3.3 提示词设计

V0.5 的提示词不是简单的"帮我写个摘要"，而是针对每个场景做了结构化设计：

**摘要**：要求核心观点一句话 + 3-5 条要点（每条 ≤30 字）+ 实践价值一句，总体 200 字以内，禁止"本文""作者"等套话开头。

**标签**：要求技术词 + 方法词 + 场景词混合，3-8 个，模拟读者搜索行为。

**润色**：明确要求修正语病、调整段落结构、首次出现的技术概念加括号解释、保留标题层级和代码块、保留作者风格不做颠覆性改写。

**大纲**：识别原文标题层级，没有标题时根据段落主题归纳 4-8 个章节名。

**标题**：生成 5 个候选——3 个务实型（直接说明内容）+ 2 个吸引型（用问题或对比引发好奇）。

**问答**：结构化为核心结论 → 展开说明 → 延伸思考；改进建议从结构、论证、可读性三个维度给出。

### 3.4 安全性设计

AI 接口的安全性设计：

1. **API Key 不写入代码**：只能通过环境变量 `AI_API_KEY` 传入。`config.py` 不含默认 Key。
2. **不配 Key 也能运行**：`ai_service.is_configured()` 检测 Key 是否存在，不存在时返回友好提示而不是报错。
3. **所有调用记录日志**：每次 AI 调用写入 `AiLog` 表，包含场景、输入、输出，方便答辩时展示 AI 协作记录。
4. **CSRF 豁免**：AI 接口使用 `@csrf.exempt`，因为它们走 JSON + fetch，不通过 HTML 表单提交。
5. **环境变量隔离**：Key 只在当前终端会话有效，关闭终端即失效。

## 四、安全加固

### 4.1 SECRET_KEY

**问题**：`config.py` 中 `SECRET_KEY = "dev-secret-key-change-before-production"`——这个值不仅在代码里，而且在公开的 GitHub 仓库中如果提交上去，任何人都能伪造 session。

**修复**：优先从 `SECRET_KEY` 环境变量读取，否则用 `secrets.token_hex(32)` 随机生成 64 字符密钥。每次启动自动生成，多进程部署时通过环境变量统一。

### 4.2 CSRF 保护

**问题**：系统有 30+ 个 POST 表单（登录、注册、写文章、评论、点赞、收藏、管理后台各种操作），全部没有 CSRF token。任何外部站点都可以伪造用户提交。

**修复方案**：集成 Flask-WTF 的 CSRFProtect。

为了避免逐个修改 30+ 个模板，采用 **JS 自动注入** 方案：

1. 在 `base.html` 和 `admin/base.html` 的 `<head>` 中添加 `<meta name="csrf-token">`。
2. 通过 Jinja2 context processor 注入 `csrf_token_value`。
3. 一段小 JS 脚本自动向所有 `<form method="post">` 注入 `<input type="hidden" name="csrf_token">`。
4. 同时拦截 `window.fetch`，为所有 AJAX 请求自动添加 `X-CSRFToken` 头。

CSRF 验证失败时的用户体验：
```
表单验证已过期，请刷新页面后重试。 → 自动跳回上一页
```

### 4.3 其他安全改进

- 新增 `WTF_CSRF_SECRET_KEY` 配置，同样支持环境变量和随机生成。
- 管理后台模板从 `base.html` 独立出来（`admin/base.html`），单独注入 CSRF 保护。

## 五、代码质量改进

### 5.1 datetime.utcnow → utcnow()

Python 3.12+ 废弃了 `datetime.utcnow()`。项目当前运行在 Python 3.9 不受影响，但应做好向前兼容。

新增工具函数：
```python
def utcnow():
    """Return a naive UTC datetime compatible with SQLite, safe for Python ≥3.12."""
    return datetime.now(timezone.utc).replace(tzinfo=None)
```

影响范围：`models.py`（22 处）、`article/services.py`（2 处）、`demo_data.py`（1 处）。

### 5.2 urllib → requests

AI 服务的 HTTP 调用从 stdlib `urllib.request` 切换为 `requests` 库：

- SSL 证书验证由 certifi 包处理，不依赖系统证书
- 异常层次清晰：`Timeout`、`HTTPError`、`ConnectionError`、`RequestException`
- 请求构造从 15 行手动拼接简化为 `requests.post(url, json=payload, headers=headers)`

### 5.3 中文标题检测

文章阅读页的目录生成逻辑原来只识别人工列举的 `一、` 到 `六、`，且任何以冒号结尾的行都被误判为标题。

改为正则匹配：
- `_CN_NUMBERED`：匹配一至二十的序号（`一、` ~ `二十、`）
- `_CN_LABEL`：匹配 2-8 个汉字后跟中文冒号的短标签

### 5.4 依赖声明

`requirements.txt` 从 3 个直接依赖补全到 8 个，全部固定版本号，避免将来安装时版本漂移。

## 九、V0.5 整体反思

### 做得好的

1. **AI 功能是可配置的，不是强绑定的**。这对课程项目很重要——评审老师可能没有 API Key，但不影响系统运行。
2. **CSRF 保护的实现方案**。用 JS 自动注入而不是逐个模板修改，省了大量手工工作且不容易遗漏。
3. **提示词设计**。不是简单的一句话，而是给出了具体的输出结构、字数限制、禁止事项，AI 输出质量明显提升。
4. **文档一次到位**。把两份分散的指南合并，统一路径，新增使用教程，后续维护成本大幅降低。

### 可以改进的

1. AI 功能目前没有流式输出，大段润色结果需要等几秒才能看到。后续可以加 SSE。
2. 测试脚本依赖数据库已有演示数据，如果数据库为空部分测试会跳过。后续应让测试完全自包含。
3. 阅读历史仍然用收藏列表冒充（`favorites[:5]`），应该建一个真正的 `reading_history` 表。

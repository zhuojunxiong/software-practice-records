# 第二天工作记录

## 一、背景

第一天完成了项目方向梳理和 V0.1 骨架搭建。之后经历了 V0.2（业务闭环）、V0.3（多用户专栏）、V0.4（前端 UI/UX 产品化改造）三个版本的迭代。

到 V0.4 结束时，系统已经从"管理员写、游客看"的单人博客，升级为"每个用户都可以创建专栏、发布文章、阅读和交流"的多用户知识创作平台。前端页面也建立了一套统一的视觉系统，首页、文章列表、专栏页、写文章页都不再像默认浏览器样式。

但 V0.4 的问题也很明显：

1. 页面虽好看，但**链路不完整**——登录后缺少个人空间，写文章页还是旧逻辑，作者主页和专栏详情页风格不统一。
2. 403、404、500 错误页还比较粗糙。
3. 缺少版本说明文档，后续无法形成可提交、可答辩的稳定节点。
4. 写作流程中存在真实使用问题：发布后跳转逻辑不清晰，草稿和发布的区分不够明显。

V0.4.1 的目标不是新增功能，而是**把 V0.4 的产品界面收口成一个链路完整、稳定可演示的版本**。

## 二、V0.4.1 定位

> 在现有 Flask + Jinja2 架构下，完成界面、认证、个人空间、写作链路、补充页面和状态页的统一重构，使系统具备稳定演示能力。

三个核心原则：

1. **不迁移技术栈**。继续使用 Flask + Jinja2 服务端渲染，不引入 Vue/React/FastAPI。
2. **不接入真实 AI**。AI 面板保留 UI 占位，提示"后续版本开放"，不发出远程请求。
3. **不改数据库结构**。复用 V0.3 建立的 User、Article、BlogColumn、Comment 等模型。

## 三、主要完成内容

### 3.1 首页分流

V0.4 的首页既承担产品介绍又承担内容聚合，两者混在一起不够清晰。V0.4.1 做了拆分：

| 路由 | 页面 | 用途 |
|------|------|------|
| `/` | 落地页 | 产品哲学介绍，引导用户进入阅读或创作 |
| `/home` | 内容首页 | 搜索入口、最新文章、推荐专栏 |
| `/discover` | 发现页 | 分类、标签、专栏、活跃作者等内容聚合 |

落地页的文案定位为"让写作回到思考，让阅读回到平等"，强调这不是商业化信息流，而是围绕知识创作的平等空间。

### 3.2 认证链路统一

登录和注册页面从旧版裸表单升级为 v0.4.1 视觉系统：

- 统一使用纸质感暖色背景、衬线标题、圆角卡片
- 登录支持用户名或邮箱，普通用户登录和后台管理员登录共用同一套页面逻辑
- 注册保留邀请码字段作为未来扩展点，当前不校验
- 登录成功后的跳转行为更合理：管理员进后台，普通用户进个人空间

### 3.3 个人空间链路

V0.4.1 建立了完整的个人空间：

| 路由 | 页面 | 内容 |
|------|------|------|
| `/profile` | 书房首页 | 写作入口、最近草稿、收藏、最近阅读、收到评论 |
| `/profile/archive` | 存档 | 用户的专栏和文章列表 |
| `/profile/reading` | 阅读轨迹 | 收藏列表和最近阅读占位（后续版本替换为真实阅读历史） |
| `/talk` | 清谈 | 收到的评论、发出的评论、待审核评论、互动概览 |
| `/settings` | 设置 | 修改密码、退出确认 |
| `/me` | 重定向 | 重定向到 `/profile` |

`/me` 改为重定向至 `/profile`，保持旧入口兼容。

### 3.4 写作链路优化

写作页 `/write` 从旧版的表单页升级为写作工作台：

- **标题区**：大字输入框，不设 placeholder 干扰
- **摘要区**：可选的题记/摘要输入
- **正文区**：宽敞的 textarea，鼓励沉浸写作
- **元数据面板**：分类下拉、专栏下拉、标签多选
- **双按钮**：保存草稿（draft）和发布文章（published），视觉层级分明
- **状态指示**：顶部显示"未保存/已保存/已加载"
- **未保存离开警告**：切换页面或关闭标签时弹出确认
- **右侧 AI 面板**：占位状态，提示"AI 阅读提示功能将在后续版本开放"

写作页的编辑模式复用 `/my/articles/<id>/edit` 旧路由，编辑已有文章时标题、摘要、正文自动回填。

### 3.5 补充页面和状态页

**作者主页**（`/users/<id>`）：

- 显示作者头像占位、昵称、简介
- 统计文章数和专栏数
- 列出该用户的专栏和已发布文章
- 不做关注、私信、社交指标

**专栏详情页**（`/columns/<id>`）：

- 显示专栏名称、描述、创建者、文章数
- 列出该专栏下的所有已发布文章
- 非活跃专栏对非创建者返回 404

**错误页面**（403、404、500）：

- 统一 v0.4.1 视觉风格
- 不显示堆栈信息
- 提供返回首页、搜索、个人空间等导航

## 四、技术决策

### 为什么继续用服务端渲染

V0.4.1 明确**不做前后端分离**。原因：

1. 课程项目的核心目标是演示软件工程过程，不是炫耀前端技术栈。
2. Jinja2 + Flask 的路由模型可以直接对应需求文档中的页面清单，答辩时容易解释。
3. 前后端分离会引入联调成本、跨域问题、构建工具链，对于这个体量的项目得不偿失。
4. V0.5/V1.0 可以先整理 JSON API 层，再从容迁移前端架构。

### 为什么 AI 继续用占位

V0.4.1 的 AI 面板是 UI 占位，不接入真实 AI。原因：

1. AI 接口依赖 API Key 管理，直接接入会增加环境配置复杂度。
2. Mock 阶段应先确保 UI 交互流程正确，后续再替换为真实服务。
3. AI 日志结构已经设计好（AiLog 表），但 V0.4.1 不需要写数据进去。
